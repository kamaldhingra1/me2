## Real-World AI Attack Examples and Control Mapping

The following examples are written as realistic enterprise attack scenarios. They are not intended as implementation instructions; they are designed to help security architects, AI platform teams, and CISOs understand how AI-specific threats unfold and how the reference architecture mitigates them.

<Callout type="warning" title="Architectural Use Only">
These scenarios are defensive examples for threat modeling, control validation, red teaming scope definition, and architecture review. The focus is on attack paths, business impact, and mitigations.
</Callout>

### Example 1: Indirect Prompt Injection Against a RAG Copilot

**Scenario:**  
An enterprise deploys a customer support copilot that uses RAG to retrieve information from internal knowledge articles, product documentation, and support tickets. An attacker places malicious instructions inside a document that later becomes part of the retrieval corpus.

**Attack path:**

```text
Malicious document uploaded
        ↓
Document indexed into RAG source
        ↓
User asks a normal support question
        ↓
RAG retrieves poisoned content
        ↓
Prompt context includes hidden malicious instruction
        ↓
Model follows attacker-controlled instruction
        ↓
Sensitive support data appears in response
```

**Primary risks:**

- Indirect prompt injection
- Retrieval poisoning
- Sensitive data leakage
- Loss of response integrity
- Trust erosion in enterprise copilots

**Mitigations from the architecture:**

- RAG source validation
- Retrieval trust scoring
- Content provenance
- Chunk validation
- Prompt/context isolation
- Output guardrails
- DLP on generated responses
- AI SOC detection for abnormal response behavior

---

### Example 2: Agent Tool Abuse Through Excessive Permissions

**Scenario:**  
A business operations agent has access to ticketing, email, CRM, and workflow automation tools. The agent is designed to summarize cases and recommend next actions. Because tool permissions are too broad, a manipulated prompt causes the agent to perform actions beyond the intended business process.

**Attack path:**

```text
User or document injects malicious task instruction
        ↓
Planner agent interprets instruction as valid workflow step
        ↓
Tool agent invokes privileged enterprise API
        ↓
Agent modifies record, sends message, or retrieves restricted data
        ↓
Action is logged but not blocked in real time
```

**Primary risks:**

- Tool abuse
- Excessive agent permissions
- Unauthorized business action
- Change-management bypass
- Privilege escalation through automation

**Mitigations from the architecture:**

- Fine-grained agent permissions
- Dynamic tool authorization
- Tool allowlisting
- Context-aware policy enforcement
- Human approval gates for high-impact actions
- Agent trust scoring
- Workflow policy enforcement
- Kill switch and circuit breakers

---

### Example 3: Memory Poisoning in a Multi-Agent Workflow

**Scenario:**  
A multi-agent research system uses shared memory to preserve findings, intermediate reasoning, and task state. A malicious or compromised input causes false information to be stored in shared memory. Other agents later rely on that poisoned memory to make recommendations.

**Attack path:**

```text
Malicious input enters agent workflow
        ↓
Planner agent summarizes false instruction or fabricated fact
        ↓
False content is written to shared memory
        ↓
Research agent retrieves poisoned memory later
        ↓
Executor agent acts on compromised context
        ↓
Business decision is influenced by poisoned state
```

**Primary risks:**

- Memory poisoning
- Context manipulation
- Cross-agent contamination
- Integrity failure in autonomous workflow
- Incorrect business decisions

**Mitigations from the architecture:**

- Memory integrity checks
- Sensitive memory tagging
- Memory segmentation by task and agent
- Memory retention and TTL policies
- Context validation before reuse
- Agent-to-agent authorization
- Continuous threat validation
- AI assurance evidence capture

---

### Example 4: Agent Impersonation and Cross-Agent Privilege Escalation

**Scenario:**  
A multi-agent architecture includes planner agents, executor agents, retrieval agents, and approval agents. An attacker attempts to impersonate a trusted agent or manipulate a lower-privilege agent into influencing a higher-privilege agent.

**Attack path:**

```text
Rogue or compromised agent joins workflow
        ↓
Agent presents unverified identity or reused credential
        ↓
Trusted agent accepts messages without strong verification
        ↓
Lower-privilege agent influences privileged workflow
        ↓
High-impact action is executed without proper authorization
```

**Primary risks:**

- Agent spoofing
- Agent impersonation
- Cross-agent privilege escalation
- Unauthorized delegation
- Autonomous action misuse

**Mitigations from the architecture:**

- SPIFFE/SPIRE-based agent identities
- Agent certificates and short-lived tokens
- Mutual TLS for agent-to-agent communication
- Signed agent messages
- Trust registry and revocation lists
- Dynamic Agent Trust Engine
- Permission inheritance controls
- Human approval gates

---

### Example 5: MCP Server or Plugin Compromise

**Scenario:**  
An AI application uses MCP servers and plugins to access enterprise systems. A malicious or compromised MCP endpoint returns manipulated tool descriptions, unsafe instructions, or responses that influence the agent to call unauthorized functions.

**Attack path:**

```text
Compromised MCP server is registered or discovered
        ↓
Agent receives manipulated tool metadata or response
        ↓
Planner selects unsafe tool action
        ↓
Tool executes against enterprise service
        ↓
Data is modified, exposed, or exfiltrated
```

**Primary risks:**

- MCP compromise
- Tool poisoning
- Unsafe tool invocation
- Data exfiltration
- Unauthorized enterprise system access

**Mitigations from the architecture:**

- MCP server allowlisting
- Signed MCP endpoints
- Tool attestation
- Tool risk scoring
- Dynamic tool authorization
- Tool behavior monitoring
- Egress controls
- AI SOC alerts for unusual tool activity

---

### Example 6: Model Supply Chain Compromise Through Malicious Adapter

**Scenario:**  
A team downloads a model adapter or fine-tuned component from an unverified source. The adapter appears to improve performance but introduces hidden behavior that activates under specific prompts or contexts.

**Attack path:**

```text
Unverified adapter added to model pipeline
        ↓
Adapter passes basic functional tests
        ↓
Model is promoted into pre-production
        ↓
Trigger prompt activates hidden behavior
        ↓
Model leaks data, bypasses policy, or produces unsafe output
```

**Primary risks:**

- Malicious adapter
- Model backdoor
- Supply chain compromise
- Model integrity failure
- Unsafe production behavior

**Mitigations from the architecture:**

- MBOM and model lineage
- Adapter provenance verification
- Artifact signing
- SLSA/Sigstore attestation
- Model scanning and adversarial testing
- AI red teaming before promotion
- Policy gates in CI/CD for AI
- Continuous runtime monitoring

---

### Example 7: Model Extraction Against an Enterprise AI API

**Scenario:**  
An enterprise exposes a model endpoint through an internal or partner-facing API. An attacker repeatedly queries the model to approximate its behavior, extract proprietary logic, or infer sensitive training information.

**Attack path:**

```text
Attacker obtains API access or misuses valid access
        ↓
High-volume probing begins
        ↓
Responses are collected and analyzed
        ↓
Model behavior is approximated
        ↓
Proprietary model logic or sensitive patterns are exposed
```

**Primary risks:**

- Model extraction
- Model theft
- Intellectual property loss
- Training data inference
- Competitive exposure

**Mitigations from the architecture:**

- Strong API authentication
- Rate limiting and throttling
- Behavioral analytics
- Output monitoring
- Model watermarking
- Differential privacy where appropriate
- Anomaly detection for probing behavior
- AI SOC investigation workflow

---

### Example 8: Autonomous Workflow Runaway Loop

**Scenario:**  
An autonomous workflow uses multiple agents to detect, triage, and remediate operational incidents. A misconfiguration or adversarial input causes agents to repeatedly invoke each other and initiate actions without convergence.

**Attack path:**

```text
Unexpected input triggers autonomous workflow
        ↓
Planner agent delegates task to executor agent
        ↓
Executor generates new tasks for planner
        ↓
Agents recursively invoke each other
        ↓
Workflow creates excessive actions or operational disruption
```

**Primary risks:**

- Recursive agent failure
- Runaway automation
- Operational disruption
- Cost explosion
- Unauthorized repeated actions

**Mitigations from the architecture:**

- Loop and recursion prevention
- Escalation limits
- Workflow policy enforcement
- Quotas and execution budgets
- Checkpointing and rollback
- Human approval gates
- Agent quarantine mode
- Kill switch and circuit breakers
